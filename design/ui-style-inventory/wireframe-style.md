# Wireframe Style

## Direction
Use the wireframe style as a calm, low-fidelity design language for shaping page behavior before committing to full visual design. It should feel like an AHA page system already exists: white-first, spatial, tactile, lightly dimensional, and edited. The goal is not a generic skeleton screen; it is a readable planning artifact that shows hierarchy, routing, content role, and emotional temperature without pretending the final art direction is solved.

## Reasoning
- The current mock-up already carries several AHA refresh principles: bright canvas, floating navigation, strategic red, human warmth, and enough spatial depth to feel modern without becoming decorative.
- Low-fidelity wireframes need more brand behavior than gray boxes alone, because the homepage question is partly about trust, clarity, and how AHA presents itself as a guide.
- Keeping the style restrained lets us compare structure, content priority, and route behavior without getting trapped in image, color, or typography debates too early.
- The wireframe should still make important UX decisions visible: what is primary, what is secondary, how people move, where search lives, and how engagement is staged.

## Style Rules
- Use a white or very light warm-gray page canvas.
- Keep the page vertically generous, with clear bands of intent rather than dense stacked modules.
- Use soft neutral gray placeholders for images, cards, proof bands, text bars, and content blocks.
- Use AHA red only for the logo, Donate, and an occasional primary action signal.
- Allow full-bleed AHA red only for the final closing section.
- Keep navigation floating in a pill-like group with a separate search control.
- Use subtle shadows and soft corners to make components feel tactile.
- Keep typography implied through gray bars, except for key labels that need to be readable for critique.
- Make large modules specific by role, not by final copy volume.
- Keep illustration placeholders soft, human, and restrained; avoid detailed final artwork at this stage.

## Cognitive Rules
- Design each viewport around one main job.
- Show one main CTA at most in any viewport.
- Show no more than three information units in any viewport.
- Use fourth-grade-simple labels wherever possible: short words, plain verbs, and direct user intent.
- Reveal content progressively through cards, drawers, and scroll beats instead of exposing the whole ecosystem at once.
- Treat simplicity as a strategic quality, not a lack of design.

## Layout Behavior
- First screen should show navigation, hero meaning, and immediate route choices.
- Three-card rows should represent primary routes or major user decisions.
- Full-width bands should represent proof, mission, or transition moments.
- Finder modules should look more interactive than passive content shelves.
- Lower-page modules should alternate between routing, proof, action, and story so the page does not become a feed.
- Use whitespace as hierarchy, not as empty leftover canvas.
- Let the bottom red heartbeat section act as the strongest color shift on the page.

## Annotation Rules
- Use numbered annotation bubbles when the wireframe needs critique notes.
- Keep annotation copy short: `1 main CTA`, `3 info units max`, `drawer opens here`, `scroll text activates`.
- Use thin leader lines and side notes, not heavy overlays.
- Annotate viewport behavior and interaction states more than decorative details.
- When text rendering is unreliable in generated images, use numbers in the image and keep the explanation in the companion documentation.

## Labeling Rules
- Use real labels for modules and primary actions when they affect the conversation.
- Use gray text bars for body copy and supporting detail.
- Do not fill the wireframe with placeholder lorem ipsum.
- Do not over-label obvious structural blocks.
- Keep labels short enough to survive image generation without unreadable clutter.

## Expansion Kit
- `floating nav`: logo, core routes, Donate, separate search.
- `hero guidance`: human image or illustration placeholder plus mission/headline/action.
- `route cards`: three large cards for primary audience or task routes.
- `proof band`: horizontal trust or mission cue.
- `search finder`: large input, suggested chips, topic shortcuts.
- `personalized filters`: three simple controls that change a drawer below.
- `expanded drawer`: recommended next step, three links, supporting story/tool/guide.
- `guide grid`: structured topic or chapter entry points.
- `feature module`: one highlighted timely guide or program.
- `action ladder`: lighter-to-heavier engagement path.
- `professional band`: compact science or professional-resource lane.
- `story proof`: human or local relevance module tied to an action.
- `scroll story text`: one large sentence revealed in stages.
- `red heartbeat close`: full-bleed AHA-red section with background heartbeat line.
- `footer recovery`: grouped links and recovery routes.

## Image-Generation Rules
- Treat the screenshot as a style and proportion reference, not as a locked layout.
- Ask for `low-fidelity homepage wireframe`, not high-fidelity web design.
- Ask for grayscale placeholders with small AHA-red accents.
- Require readable module labels but gray bars for body copy.
- Avoid final photography, glossy UI, heavy red, dark hero sections, gradients, decorative blobs, and generic stock healthcare imagery.

## Open Questions
- Should the homepage wireframe use an open search field in the header, a large hero search, or both?
- Should route cards prioritize audience groups, user tasks, or a hybrid of both?
- How much real sample copy should appear before the wireframe starts behaving like a comp?
- How many annotation bubbles can a generated wireframe support before the image becomes harder to read?
